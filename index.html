PARMco.

Where the rind meets the road. A BLE-connected motor-control system linking an iOS app to a Raspberry Pi 4 for real-time RPM control, feedback, and mode switching.

Est. Spring 2026 Athens, GA Final Project iOS ↔ RP4 ↔ Motor
01 · The Project

A final project with flavor.

PARMCO stands for Phone App RP4 Motor Control. The project was built to satisfy the ECSE 4235 requirement of a standalone system in which a phone app connects over Bluetooth to a Raspberry Pi 4, controls a DC motor, and reports live RPM back to the user. The final system supports direction control, start and stop, manual speed control, automatic RPM maintenance, and a match mode that tracks a second motor using a separate optical sensor.

This site is the public face of that work. The repository contains the iOS app, the Raspberry Pi control software, the assembly GPIO helpers, documentation, and iteration history. The goal was not just to make the motor spin, but to build a clean end-to-end system that could be demonstrated, reinstalled, re-paired, and understood by someone reading the code cold.

02 · System Overview

One app. One Pi. One control loop.

Step 01

iOS App

SwiftUI provides the front end for scanning, pairing, mode selection, motor start/stop, direction changes, speed updates, and target RPM entry.

Step 02

BLE Link

The phone discovers the Pi as PARMCO-RP4, connects over BLE, writes command packets, and subscribes to notification characteristics for RPM feedback.

Step 03

RP4 Control

The Raspberry Pi 4 runs a C-based control program that parses commands, updates state, drives GPIO, and maintains the motor-control threads.

Step 04

Physical Output

The motor driver stage applies PWM and direction signals while IR sensors feed rotational data back into the software for reporting and automatic control.

03 · What's Inside

Built in four layers.

/ 01

Interface

A SwiftUI front end organizes connection state, direction toggles, manual speed control, automatic RPM input, live telemetry, and reconnect behavior in a single shared Bluetooth session.

/ 02

BLE Stack

Custom service and characteristic UUIDs define the command path, motor RPM feedback path, and matched RPM feedback path. The app writes commands and subscribes to notifications.

/ 03

Embedded Control

The Raspberry Pi code manages advertising, ATT/L2CAP setup, command parsing, multithreaded motor timing, RPM measurement, and control-mode switching.

/ 04

Hardware

The hardware layer includes the motor, MOSFET-driven output stage, propeller-based optical feedback, and direct GPIO manipulation through ARM assembly helper routines.

04 · Technical Architecture

The real system under the branding.

App-side behavior

The iOS app uses a shared BluetoothManager object so the discovery UI and motor-control UI operate on the same active BLE session. After connect, the app pushes the current UI state back to the Pi so reconnects do not leave the embedded side stale. Slider-based control writes are debounced to reduce command spam while keeping the interface responsive.

Pi-side behavior

The Raspberry Pi 4 advertises as PARMCO-RP4, accepts BLE connections, instantiates a GATT server, and runs independent threads for motor drive, primary RPM measurement, and matched-RPM measurement. The code stores control variables atomically so command updates from BLE and sensor updates from the worker threads remain synchronized.

  • GPIO 23 / 24: motor direction outputs
  • GPIO 18: primary IR sensor for local RPM
  • GPIO 22: second IR sensor for match mode
  • GPIO 27: BLE status LED
  • PWM base rate: 1 kHz
  • RPM window: 200 ms

05 · Control Modes

Manual when you want it. Closed-loop when you need it.

/ 01

Manual

The user directly chooses a duty-cycle-style speed level from the app. This mode is useful for baseline testing, verifying direction changes, and measuring how the motor behaves without automatic correction.

/ 02

Auto / Maintain

The app provides a target RPM and the Pi continuously compares that target to measured RPM from the primary IR sensor. A feedforward estimate is combined with PID trim to keep the motor near the requested speed.

/ 03

Match

A second IR sensor measures another spinning propeller. The Pi treats that reading as the effective target and adjusts its own duty cycle so the controlled motor follows the other motor’s RPM in real time.

06 · Implementation Notes

Why this build is technical.

Signal and timing decisions

  • RPM is measured by counting debounced optical pulses in fixed windows rather than trusting a single edge.
  • The code assumes three pulses per revolution, which is then scaled into RPM.
  • Very low or noisy pulse counts are rejected to avoid reporting false rotation.
  • Automatic control uses filtered RPM values instead of raw quantized readings to stabilize updates.

Software architecture decisions

  • The BLE session is preserved across views by sharing one manager instance through the SwiftUI app.
  • The phone remembers the Pi UUID and can forget and re-pair when needed.
  • The Pi starts in a quiet state and waits for the phone, matching the project specification.
  • GPIO helper functions are implemented in ARM assembly for direct register-level control on the BCM2711.
